Sistema de Información
A. Definición del problema o problemas que resuelve el Sistema de
Información, u oportunidad de negocio.
Problemas actuales:
• La asistencia se registra en papel/Excel, lo que provoca errores,
duplicidad de datos y retrabajo.
• No existe un flujo formal de permisos/ausencias con aprobación, por lo
que hay poca trazabilidad (quién autorizó, cuándo y por qué).
• El reporte mensual para planilla (tardías, faltas, permisos
aprobados/rechazados) se arma manualmente y puede salir
inconsistente.
• No se tiene una vista sobre el perfil del usuario.

Oportunidad de negocio:
• Reducir tiempo de RRHH en tareas repetitivas, mejorar exactitud de
planilla y tener control interno.

B. Fijar los objetivos del sistema.
Objetivo general:
Implementar un Sistema de Información que registre asistencia, gestione
permisos con aprobación y genere reportes mensuales confiables para
planilla.
Objetivos específicos:
• Gestionar permisos/ausencias con estados (pendiente, aprobado,
rechazado) y comentarios.
• Mantener trazabilidad de acciones (aprobaciones, cambios, ajustes).
• Generar reportes mensuales exportables (Excel) para RRHH/planilla.
• Controlar accesos con roles (Empleado, Supervisor, RRHH/Admin).
• Visualizar perfiles de los empleados.
C. Identificación de las restricciones del Sistema de Información.
• Tamaño de operación:
Se considera que la empresa cuenta con 30 empleados, por lo tanto, debe
ser una solución “ligera” sin incurrir en costos altos.
• Conectividad:
El sistema debe funcionar en red local o internet estable; si se cae internet,
se manejará la política “Solo en línea”, la cual indica que, si no hay
conexión, no se permite marcar y podrá notificar a RRHH, o esperar a que
haya conexión nuevamente.
• Presupuesto y mantenimiento:
Se priorizan herramientas open source y despliegue simple
(Docker/servidor Linux).
• Adopción del usuario:
interfaz intuitiva con la finalidad de que el usuario pueda marcar y solicitar
permiso en pocos clics.
• Seguridad y privacidad:
control de acceso por roles, contraseñas seguras, bitácora de acciones,
respaldos.
• Calidad de datos inicial:
migración de Excel puede venir con inconsistencias, por lo que, se deberá
realizar una depuración mínima.



D. Definición de la plataforma de Sistema Operativo y su justificación.
El sistema se desarrollará como una aplicación web multiplataforma, accesible
desde los principales navegadores disponibles en Windows, Linux y entornos
Apple (macOS/iOS), sin depender de un sistema operativo específico para los
usuarios.
Justificación:
• Compatibilidad en distintos sistemas operativos: Navegador de soporte
oficial: Google Chrome (última versión estable - 5); ejecución compatible
también con Edge, Firefox y Safari en Windows/Linux/macOS.

• Sin instalaciones complejas: Los usuarios no deben instalar un programa
por sistema operativo; únicamente ingresan al sistema mediante una URL
desde su navegador.

• Menor mantenimiento y mayor adopción: Al centralizar el uso en el
navegador, se reducen problemas de compatibilidad por versiones del SO y
se facilita el soporte (RRHH/empleados solo necesitan un navegador
actualizado).

• Basado en estándares web: El sistema utilizará tecnologías estándar, lo
que asegura interoperabilidad entre navegadores modernos y equipos
heterogéneos.

• Escalabilidad de acceso: Permite usar computadoras de escritorio o
laptops (Windows/Linux/macOS) e incluso dispositivos móviles
(iOS/Android) con diseño responsivo.

E. Definición del Sistema Gestor de Base de Datos y su justificación.

Gestor definido: Microsoft SQL Server
Justificación:
• Alineación con ecosistema Microsoft: si la organización ya utiliza
herramientas Microsoft, SQL Server se puede integrar muy bien y facilita la
administración y el análisis de datos.
• Robusto para transacciones y auditoría: es adecuado para datos
transaccionales como permisos, aprobaciones y bitácora, manteniendo
consistencia e integridad y soportando consultas para reportes mensuales.
• Seguridad y control: ofrece mecanismos sólidos de roles, permisos,
cifrado, auditoría y respaldos, importantes para información de RRHH.
• Herramientas de administración y mantenimiento maduras: facilita
tareas como monitoreo, respaldos, restauración y optimización, reduciendo
riesgos operativos.
• Escalabilidad: aunque hoy son 30 empleados, SQL Server puede crecer sin
cambiar de motor; solo se ajusta la edición y recursos cuando sea
necesario.
F. Definición de la Herramienta de Desarrollo y su justificación.
Stack:
• Frontend: Angular (SPA) para pantallas del sistema (login, marcaje,
permisos, reportes).
• Backend: NestJS (Node.js más TypeScript) para la API REST y lógica de
negocio (autenticación, roles, reglas de tardías, aprobaciones,
reportes).
• Base de datos: Microsoft SQL Server para el almacenamiento
transaccional (asistencia, permisos, auditoría).
Justificación:
• Estandarización en TypeScript: Angular (frontend) y NestJS (backend)
comparten TypeScript, reduciendo curva de aprendizaje y facilitando
mantenimiento del código.
o Arquitectura modular y escalable: NestJS trabaja por módulos
(usuarios, asistencia, permisos, reportes), lo cual se adapta bien al
crecimiento por funcionalidades.
o Compatibilidad multiplataforma: al ser una aplicación web, funciona
desde navegadores en Windows, Linux y entorno Apple, sin
instalaciones adicionales en los equipos del usuario.
o Robustez en datos y reportes: SQL Server es un motor maduro para
operaciones transaccionales y generación de reportes mensuales de
planilla.
o Desarrollo controlado y trazable: el ORM permite centralizar modelos,
validaciones y migraciones, reduciendo errores y facilitando cambios
en el esquema.

G. Metodología de Desarrollo que se utilizará y su justificación.

Metodología de desarrollo: Ágil (Scrum).
Justificación:
• Serán sprints de 2 semanas porque se estará llevando a cabo por
tiempos libres. También se tomó en cuenta porque es un proyecto
mediano y conviene tener avances visibles rápido, corregir temprano y
no acumular cambios.
• Requerimientos como reglas de tardías, tipos de permisos y formato de
reportes suelen ajustarse con RRHH al ver prototipos.
• Permite entregar valor rápido: primero marcaje, luego permisos, luego
reportes, y al final mejoras.

H. Herramienta o Mecanismo de Control del Proyecto
Herramienta definida: Azure DevOps para el seguimiento y control del proyecto.

Justificación:
• Scrum requiere gestionar un Product Backlog y Sprint Backlog, así como
dar visibilidad al avance; Azure DevOps permitirá organizar y priorizar
historias de usuario, tareas y entregables por sprint.
• Facilita el control del trabajo mediante tableros (Scrum/Kanban), lo que
apoya la ejecución de los eventos de Scrum: Planning (selección de
trabajo del sprint), Daily (seguimiento rápido del
estado), Review (validación de incrementos) y Retrospective (mejora
continua).
• Permite mantener trazabilidad del progreso y evidencia del proyecto
(estados, comentarios y registro de incidencias), alineándose con el
enfoque iterativo e incremental de Scrum.
Dirección de Proyecto
A. BUSINESS CASE
Nombre del Proyecto: Sistema de Control de Asistencia y Permisos
• Problema/Oportunidad: Registro manual de asistencia en papel/Excel causa
errores, duplicidad y retrabajo. No hay flujo formal de permisos ni trazabilidad.
Reportes mensuales inconsistentes.
• Objetivos: Automatizar registro de asistencia, gestionar permisos con
aprobación, generar reportes confiables, reducir tiempo de RRHH.
• Justificación: Reducción de errores en planilla, ahorro de tiempo
administrativo, mayor control interno, trazabilidad de aprobaciones, mejora en
la toma de decisiones.
• Alcance: Gestionar permisos/ausencias con estados (pendiente, aprobado,
rechazado) y comentarios, Mantener trazabilidad de acciones (aprobaciones,
cambios, ajustes), Generar reportes mensuales exportables (Excel) para RRHH,
Controlar accesos con roles (Empleado, Supervisor, RRHH/Admin), y Visualizar
perfiles de los empleados.
• Costo Estimado: Bajo (open source).
• Beneficios Esperados: 30% reducción en tiempo de procesamiento de planilla,
eliminación de errores manuales, trazabilidad completa, mejora en
cumplimiento normativo.
• Riesgos Principales: Resistencia al cambio, inconsistencia en migración de
datos, caída de conectividad.
• Criterios de Éxito: Sistema operativo en 4 meses, adopción del 90% de
empleados, reportes mensuales generados sin errores.
B. ENUNCIADO DEL PROYECTO
Desarrollar e implementar un sistema web de control de asistencia y gestión de
permisos para Servicios Integrales Maya, que reemplace el registro manual en
papel/Excel, automatice el flujo de aprobaciones, genere reportes confiables y ofrezca
trazabilidad completa de las acciones, mejorando la eficiencia de RRHH y la exactitud
de los datos.
C. ALCANCE DEL PROYECTO
Incluye:
• Módulo de solicitud y aprobación de permisos/ausencias con estados
(pendiente, aprobado, rechazado) y comentarios.
• Roles de usuario: Empleado, Supervisor, RRHH/Admin.
• Generación de reportes mensuales exportables a Excel (tardías, faltas,
permisos).
• Bitácora de auditoría (trazabilidad de aprobaciones y cambios).
• Perfiles de empleados con información básica.
• Interfaz web responsiva, compatible con navegadores modernos.
• Migración asistida de datos históricos desde Excel.
Excluye:
• Integración con sistema de planilla externo.
• App móvil nativa (solo web responsiva).
• Notificaciones por SMS.

Propósito / Justificación del proyecto
Actualmente la empresa registra ausencias en formatos manuales y Excel, lo que
genera errores, retrabajo y falta de trazabilidad en permisos. El proyecto SCAP busca
digitalizar el control de asistencia y permisos para mejorar la exactitud de la
información, fortalecer el control interno y optimizar la generación del reporte
mensual para planilla.
Objetivos:
1. Automatizar el registro de asistencia de aproximadamente 30
empleados (entrada/salida e historial).
2. Implementar un flujo digital de solicitud y aprobación/rechazo de
permisos con trazabilidad (quién aprobó, cuándo y observación).
3. Generar reportes mensuales automáticos para planilla (tardías, faltas y
permisos aprobados/rechazados).
4. Reducir en al menos 30% el tiempo dedicado a tareas manuales de RRHH
relacionadas con consolidación de asistencia y permisos.
Alcance:
• Registro de usuarios y roles: Empleado, Supervisor, RRHH/Admin.
• Gestión de permisos/ausencias con estados: pendiente, aprobado, rechazado,
incluyendo comentario.
• Bitácora básica de acciones (aprobaciones, ajustes autorizados).
• Reporte mensual para planilla y exportación (según necesidad: Excel).
• Capacitación básica para RRHH, supervisores y empleados
Fuera de alcance (excluye)
• Cálculo de nómina/planilla completa (solo se entrega reporte para planilla).
• Integración con sistemas externos (ERP, biométricos, reloj físico) (a menos que
se solicite formalmente como cambio de alcance).
• Aplicación móvil nativa (Android/iOS) (se cubre por web responsive si aplica).
Criterios de Aceptación:
• Sistema operativo en ambiente de producción y estable (sin fallas críticas).
• Al menos 90% de empleados registrando asistencia en el sistema durante el
periodo de adopción.
• Flujo de permisos funcionando con trazabilidad completa
(aprobación/rechazo con responsable y fecha).
Riesgos Principales:
• Migración de datos con errores: depuración previa, carga controlada,
validación por RRHH.
• Resistencia de usuarios: capacitación breve, comunicación clara y piloto
inicial.
• Caídas de conectividad: política definida + procedimiento de contingencia
documentado
